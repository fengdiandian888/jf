## 🔴 阻断项（影响长期维护和稳定性）

### 1. CSS 变量定义四份，且不一致

**文件：** 全部四个文件

每个文件都自己定义了一套 `:root` 变量，但定义内容不同：

| 变量 | 今日计划 | 全面计划 | 跟练 | 清单 |
|------|---------|---------|------|------|
| `--warn` | ❌（用 `--warning`） | ✅ | ✅ | ✅ |
| `--danger` | ✅ | ✅ | ✅ | ❌（用 `--red`） |
| `--recovery` | ✅ | ✅ | ✅ | ❌（用 `--accent`） |
| `--sleep` | ✅ | ❌ | ❌ | ❌ |

**问题：** 同一个颜色在不同文件里有不同变量名。改一个全局色需要改 4 个文件，且容易漏。

🔧 **建议：** 提取一个 `shared.css` 作为唯一变量来源，四个页面引用同一份。如果不能拆文件，最少要做一次全域变量名归一化。

---

### 2. 核心数据硬编码多处，无单一事实源

**文件：** 全部四个文件

以下数据在至少 2 个文件中独立硬编码：

- 蛋白质克数：27g / ~20g / 53g → 100g（4个文件都写了）
- 饮水 5×400ml → 2.5L（4个文件）
- 训练动作清单（拉日 4 动作、推日 4 动作、腿日 4 动作）→ 全面计划页和跟练页各自写一份
- B站 BV 号：BV1WM411U7yq（肩康复）在 4 个文件中出现了 **14 次**
- 周日备餐 5 步骤 → 全面计划页和清单页各自写一份
- 里程碑 85→80→75→70→65 → 全面计划页和清单页各一份

**问题：** 改一个数据（比如蛋白质从 98g 变成 100g）需要在 4 个地方改。漏一处就出现数据不一致。

🔧 **建议：** 最低限度——用 JS `const` 对象集中定义所有共享数据（蛋白、饮水、动作、BV号），然后通过 HTML 内联引用。理想方案：一个 `data.js` 作为单一数据源，四个页面都引它。

### 3. 组件重复实现（Tab Bar / Bottom Nav / 天气）

**文件：** 全部四个文件

以下 UI 组件在四个文件中**完全重复**实现：

- **Tab 栏**（今日/计划/跟练/清单）：4 份完全相同的 HTML + CSS
- **底部导航**：与 Tab 栏 95% 相同的 HTML + CSS（只是激活态不同）
- **天气卡片**：今日计划和全面计划各写了一个独立的天气请求

**具体的代码重复量：**

| 组件 | 每份行数 | 出现次数 | 累计浪费 |
|------|---------|---------|---------|
| Tab 栏 CSS | ~12 行 | 4 次 | 36 行冗余 |
| Tab 栏 HTML | ~6 行 | 4 次 | 18 行冗余 |
| Bottom Nav CSS | ~6 行 | 4 次 | 18 行冗余 |
| Bottom Nav HTML | ~4 行 | 4 次 | 12 行冗余 |


### 5. 错误处理过于宽松

**文件：** 今日计划.html（行 641）、全面计划.html（行 565）、跟练.html（行 851）

三个文件的 Service Worker 注册全部静默吞错：

```javascript
navigator.serviceWorker.register('/jf/serviceWorker.js').catch(function() {});
```

**问题：** 如果 SW 注册失败（路径错误、浏览器不支持、网络问题），用户和开发者都不知道。而且三个文件各写了一份。

🔧 **建议：** 
```javascript
navigator.serviceWorker.register('/jf/serviceWorker.js').catch(function(err) {
  console.warn('PWA Service Worker 未注册：', err.message);
});
```

### 6. 硬编码的天气 API URL 无降级

**文件：** 今日计划.html（行 630）、全面计划.html（行 538）

```javascript
fetch('https://wttr.in/Zhuzhou?format=j1')
```

**问题：**
- wttr.in 是免费公共服务，无 SLA 保障
- 如果挂了，页面只显示骨架屏，用户看到的永远是"加载天气..."
- 今日计划用了 localStorage 缓存+降级逻辑，全面计划没有

🔧 **建议：** 统一天气获取逻辑。至少加一个 timeout 和友好降级提示。两个文件应该调用同一份代码。

### 7. 跟练页 `switchTab` 函数的脆弱选择器

**文件：** 全面计划.html（行 556-560）

```javascript
function switchTab(name) {
  document.querySelector(`[onclick="switchTab('${name}')"]`).classList.add('active');
```

**问题：** 用 `onclick` 属性值做选择器。如果哪天修改了调用方式（比如改成 `addEventListener`），这个选择器就会静默失效，没有任何报错。

🔧 **建议：** 改用 `data-` 属性：
```html
<button class="tab-btn pull active" data-tab="pull">
```
```javascript
document.querySelector(`[data-tab="${name}"]`).classList.add('active');
```

### 8. 跟练页视频自动加载逻辑有竞态

**文件：** 跟练.html（行 843-847）

```javascript
setTimeout(function(){
  document.querySelectorAll('.vid-lazy[data-bvid]').forEach(function(el){
    loadVid(el);
  });
}, 600);
```

**问题：**
- 600ms 硬编码延迟，慢网络可能还没加载完 DOM
- 22 个 iframe 同时加载会瞬间发起 22 个 HTTP 请求
- 如果任一 BV 号失效，iframe 会显示 B 站错误页但不报错

🔧 **建议：** 
1. 等 `DOMContentLoaded` 或 `load` 事件再批量加载
2. 用 IntersectionObserver 做可视区域懒加载（只加载屏幕内的）
3. 加 BV 号校验和加载失败提示

## 💭 优化建议（锦上添花）

四个文件都缺少 description meta 标签。微信/QQ 分享时只能看到 title 和 URL，没有描述摘要。

🔧 **建议：** 每个页面加：


### 11. 无障碍可访问性（A11y）完全缺失

四个文件在无障碍方面几乎是空白：

- 无 ARIA labels
- 无 `<main>`、`<nav>` 语义标签（仅 tab-bar 用了 nav）
- 图片/图标用的是 emoji，无 `alt` 文本
- 颜色对比度未验证（`--sub: #9ca3af` 在 `#faf8f5` 背景上，对比度约 2.1:1，低于 WCAG AA 要求的 4.5:1）

🔧 **建议：** 如果是个人使用可以降低优先级。但如果要给其他人用，至少给 `#9ca3af` 加深到 `#6b7280`。

---

### 12. 无版本号机制

四个文件改完直接 `git push`，没有内嵌版本标识。用户手机 PWA 缓存可能还是旧版。

🔧 **建议：** 在 `<head>` 加一个版本注释：
```html
<!-- v19.2 · 2026-06-24 -->
```

或者在全局 CSS 变量里定义 `--version: "19.2"`。

## 总结

**代码质量等级：C+**（功能正确但维护成本高）

四个文件当前最大的风险不是 bug，而是**数据漂移**——同一份数据散落在多个文件中，未来任何一次修改都可能引入不一致。由于这是个人项目，短期问题不大，但如果计划长期维护或交给他人接手，建议优先解决以下三项：

1. **提取共享数据源**（data.js 或 JS 全局对象）
2. **统一 CSS 变量名**（全局归一化 `--warn` / `--danger` / `--red` / `--warning`）
3. **合并天气逻辑**（两个文件用同一份代码）

这三项改造估计约 2 小时工作量，但能消除未来 80% 的改错风险。
